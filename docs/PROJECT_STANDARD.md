# MallCloud 项目开发与交付标准

> 文档状态：生效
> 适用范围：需求、设计、代码、配置、测试、部署与答辩材料
> 默认环境：Windows 11、PowerShell 7+、UTF-8、JDK 17、Maven 3.9+

---

## 1. 文档目的

本文件是 MallCloud 后续开发的统一基准，用于控制项目规模、约束技术实现、统一文档口径，并直接对应课程评分标准。

项目后续遵循以下顺序：

```text
先修改标准与需求 → 再修改设计与接口 → 再修改代码与配置 → 最后补测试证据
```

不得只修改代码而不更新相关文档，也不得把规划能力描述为已实现能力。

---

## 2. 项目定位

MallCloud 是一个面向微服务课程期末大作业的电商系统，重点展示：

- 合理的微服务划分；
- Spring Cloud Alibaba 核心组件的正确使用；
- 一条完整、可验证的交易主链路；
- 服务治理、分布式事务、消息异步和限流降级；
- 可重复执行的接口测试、负载测试、压力测试和异常测试。

项目不以功能数量为目标，不继续通过新增服务、页面或业务模块扩大规模。

---

## 3. 范围冻结

### 3.1 核心业务链路

后续开发必须优先保证以下链路完整：

```text
用户登录
  → 浏览/查询商品
  → 加入购物车
  → 创建订单
  → 查询商品信息
  → 锁定库存
  → 支付结果通知
  → 更新订单状态
  → 确认扣减库存
  → 查询订单结果
```

核心服务：

- `mall-gateway`
- `mall-auth`
- `mall-user`
- `mall-product`
- `mall-cart`
- `mall-order`
- `mall-inventory`
- `mall-pay`

### 3.2 技术亮点

以下能力完成最小闭环即可，不继续扩展业务范围：

- `mall-search`：Elasticsearch 商品搜索；
- `mall-seckill`：Redis 预扣、限购和 Sentinel 限流；
- `mall-message`：RocketMQ 消息消费与跨服务状态更新。

### 3.3 辅助服务

以下服务保持最小实现：

- `mall-admin-biz`：后台聚合查询和基础管理；
- `mall-job`：超时订单关闭、库存对账等基础任务。

### 3.4 明确不做

本期不新增或扩展：

- 真实支付渠道接入；
- 物流轨迹和配送系统；
- 优惠券、满减、促销规则引擎；
- 推荐算法；
- 客服 IM；
- 多租户；
- 完整生产级 Kubernetes 高可用集群；
- MySQL 主从切换、MHA 等生产容灾；
- 为所有 Feign Client 机械增加降级类；
- 与评分目标无直接关系的复杂兜底框架。

## 6. 架构实现标准

### 6.1 服务边界

- 每个业务服务只访问自己的业务库；
- 跨服务查询或写操作通过 OpenFeign 或 RocketMQ；
- 禁止直接连接其他服务数据库；
- 不新增服务，除非现有边界无法承载且经需求文档明确批准。

### 6.2 Gateway

- 统一入口为 `http://localhost:9000`；
- Gateway 负责路由和 JWT 基础认证；
- 白名单接口必须显式配置；
- 下游通过 `X-User-Id`、`X-User-Roles` 获取用户上下文；
- 未实现方法级权限控制前，不得在文档中声明已使用 `@PreAuthorize`。

### 6.3 Nacos

- 用于服务注册和配置中心；
- 所有 Nacos 配置必须是合法 YAML；
- YAML 注释只使用 `#`；
- 配置热更新必须通过实际测试后才能标记为已验证；
- DataId、Group、Namespace 必须在部署文档中明确。

### 6.4 OpenFeign

- 核心同步调用使用 OpenFeign；
- 调用方必须检查 `Result` 是否成功和数据是否为空；
- 仅对核心链路补充有业务意义的 fallback；
- 不要求每个辅助 Client 都创建空泛降级类。

优先治理的调用：

- `order → product`
- `order → inventory`
- `message → order`
- `message → inventory`

### 6.5 Seata

- `@GlobalTransactional` 只描述实际参与的资源；
- 当前普通下单重点验证 `mall-order` 与 `mall-inventory` 的事务一致性；
- 未参与事务的支付服务不得画入 Seata 分支；
- 必须通过失败场景验证订单和库存状态是否一致。

### 6.6 RocketMQ

- 文档必须区分普通消息和事务消息；
- 当前没有 `@RocketMQTransactionListener` 时，不得把相关 Topic 描述为事务消息；
- 消费者必须具备清晰的幂等策略或状态幂等约束；
- 不为课程项目额外实现复杂本地消息表，除非测试证明有必要。

### 6.7 Sentinel

- 只选择少量核心资源进行限流/熔断展示；
- 建议资源：创建订单、秒杀请求；
- 未使用 `@SentinelResource` 时，应按实际 Web 资源或规则来源描述；
- 必须提供一次限流和一次下游异常测试证据。

## 7. 代码质量标准

### 7.1 必须满足

- 全部 Maven 模块可编译；
- Controller、Service、Mapper 分层清晰；
- 核心写操作具有明确事务边界；
- 统一使用 `Result<T>` 和统一异常处理；
- 核心远程调用检查失败结果；
- 配置文件无语法错误；
- 不提交真实密钥、Token 或个人绝对路径；
- 不保留明显占位实现，例如固定伪 Token、`dummy` 请求体和不可执行命令；
- 关键业务日志包含订单号、请求 ID 或用户 ID 等定位字段。

### 7.2 不要求

- 不要求所有类都写大量注释；
- 不要求所有 Feign Client 都有 fallback；
- 不要求所有 Service 达到同一覆盖率；
- 不为展示设计模式而额外抽象；
- 不进行与当前缺陷无关的大范围重构。

### 7.3 测试优先级

重点测试：

1. 订单创建与库存锁定；
2. 支付结果更新订单并扣减库存；
3. 秒杀限购和库存边界；
4. Gateway JWT 鉴权；
5. 核心 Feign 失败场景。

## 8. 测试交付标准

### 8.1 Postman

必须满足课程要求：

- 核心接口不少于 6 个；
- 总请求次数不少于 20 次；
- 包含正常流和异常流；
- 登录后自动保存 Token；
- 创建订单后自动保存 `orderNo`；
- 请求体使用真实 DTO 字段；
- 至少证明一次 OpenFeign 调用成功；
- 保存 Newman 或 Postman 运行报告。

请求数量不作为额外目标，禁止使用机械复制的无效请求充数。

### 8.2 服务治理

必须验证：

- Nacos 服务注册、心跳与实例下线感知；
- Gateway 路由；
- 无 Token、错误 Token、有效 Token；
- 核心服务恢复后的重新注册。

### 8.3 JMeter

至少建立三个场景：

| 场景 | 目的 |
|---|---|
| 商品查询负载 | 50 用户与 150 用户对比 |
| 创建订单负载 | 验证 P95、吞吐量和错误率 |
| 秒杀压力 | 逐级增加到 500 用户并验证限流 |

报告至少记录：

- 并发用户；
- Ramp-up；
- 持续时间；
- 平均响应时间；
- P95；
- 吞吐量；
- 错误率；
- CPU 和内存峰值；
- 测试机器配置和代码 Commit。

性能目标以课程要求为基线：目标 P95 小于 1 秒；无法达到时如实记录瓶颈和原因，不伪造结果。

### 8.4 异常测试

至少完成：

- 一个下游服务宕机或超时场景；
- 一次 Sentinel 限流或熔断；
- 一次 Nacos 配置热更新；
- 一次库存不足或并发边界验证。

## 9. 文档质量标准

每项关键结论至少提供一种证据：

- 代码路径；
- 配置文件路径；
- 执行命令；
- 测试报告；
- 截图；
- 数据库查询结果。

以下内容禁止出现：

- 不存在的脚本或文件；
- 未执行过的“一键启动”；
- 没有报告支撑的精确性能数字；
- 把建议架构写成当前架构；
- 占位团队成员姓名；
- 失效 Token、伪业务数据和本机绝对路径。

---

## 10. 修改流程

每个任务按以下顺序执行：

1. 明确完成标准；
2. 更新 PRD 或架构/API 契约；
3. 实现最小必要代码；
4. 编译或运行验证；
5. 增加针对性测试；
6. 更新测试报告和最终报告；
7. 记录未验证项。

---

## 11. 完成定义

一个功能只有同时满足以下条件才可标记完成：

- 文档范围明确；
- 代码实现完整；
- 核心异常已处理；
- 编译通过；
- 至少一种自动化或手动验证通过；
- 相关文档已同步；
- 无占位数据和虚假结果。

---

## 12. 当前优先级

1. 修正文档与代码不一致；
2. 修复配置、构建和启动问题；
3. 打通核心交易链路；
4. 建立真实 Postman 集合；
5. 建立 JMeter 脚本和测试报告；
6. 完成服务治理与异常演示；
7. 最后整理答辩材料。
