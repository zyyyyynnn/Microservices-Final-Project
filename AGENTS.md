# MallCloud Agent 工作协议

> 本文件用于约束自动化 Agent 和开发成员执行后续任务。
> 最高标准：`docs/PROJECT_STANDARD.md`
> 默认环境：Windows 11、PowerShell 7+、UTF-8、JDK 17、Maven 3.9+

---

## 1. 必读顺序

执行任何代码、配置或文档修改前，按顺序阅读：

1. `docs/PROJECT_STANDARD.md`
2. `docs/PRD.md`
3. `docs/ARCHITECTURE.md`
4. `docs/API.md`
5. `docs/DATABASE.md`
6. `docs/CODING_STYLE.md`
7. `docs/DEPLOY.md`
8. `docs/QUICK_START.md`

测试任务还需阅读：

- `docs/test/README.md`
- `docs/FINAL_REPORT.md`

## 2. 项目状态

当前项目不是“全部就绪后只需填充业务代码”，而是：

- 微服务结构和主要代码已建立；
- 文档正在统一；
- 配置、构建和核心链路需要验证；
- Postman 集合需要重建；
- JMeter 脚本和结果尚未完成；
- Docker/Kubernetes 全栈不是当前正式交付路径。

Agent 不得假定某项能力已经完成，必须以代码、配置和运行结果为依据。

---

## 3. 总体执行顺序

```text
修复事实错误
  → 完善必要实现
  → 编译/运行验证
  → 补针对性测试
  → 更新报告
```

禁止：

- 先新增功能再修核心缺陷；
- 为扩大规模新增微服务；
- 为所有 Feign Client 机械增加 fallback；
- 引入与评分无直接关系的新框架；
- 把规划能力写成已实现；
- 使用不可执行命令、虚假 Token 或伪性能结果。

---

## 4. 任务完成标准

每个任务开始前确定：

1. 要解决的具体问题；
2. 允许修改的文件；
3. 完成后的可观察结果；
4. 验证命令或测试；
5. 需要同步的文档。

一个任务只有同时满足以下条件才算完成：

- 修改范围与需求一致；
- 项目编译通过或给出明确无法编译原因；
- 核心场景通过测试；
- 相关文档已同步；
- 未验证项明确记录；
- 未引入无关重构和额外功能。

## 5. 当前开发优先级

### P0：基础可运行

1. 修复 Nacos YAML 非法注释；
2. 统一配置加载、环境变量和 Namespace；
3. 验证数据库初始化；
4. 验证全部 Maven 模块编译；
5. 验证核心服务注册；
6. 验证 Gateway JWT。

### P1：交易主链路

1. 登录并查询用户；
2. 查询商品；
3. 购物车；
4. 创建订单；
5. 商品和库存 Feign；
6. Seata 回滚；
7. 支付消息；
8. 订单和库存状态更新。

### P2：课程技术亮点

1. Sentinel 限流/熔断；
2. 秒杀限购和库存边界；
3. Elasticsearch 搜索；
4. Nacos 热更新。

### P3：测试和答辩

1. 重建真实 Postman 集合；
2. 建立三套 JMeter 脚本；
3. 保存结果、截图和环境信息；
4. 填写 `docs/FINAL_REPORT.md`；
5. 整理演示脚本。

## 6. 代码修改规则

### 6.1 精准修改

- 只改完成任务必须修改的内容；
- 不顺手统一全项目命名；
- 不大范围重构现有模块；
- 不改变与任务无关的接口或数据库结构；
- 保持现有 Java 包和模块风格。

### 6.2 微服务边界

- 业务服务只访问自己的数据库；
- 跨服务同步调用使用 OpenFeign；
- 异步状态使用 RocketMQ；
- 不新增服务；
- 不形成循环 Feign 调用；
- 聚合服务不复制核心业务写逻辑。

### 6.3 失败处理

- 核心失败必须返回失败；
- 不通过 fallback 返回伪成功；
- 不 catch 后吞掉事务异常；
- 不使用固定业务数据掩盖下游不可用；
- 重试仅用于明确幂等操作。

### 6.4 事务

- 本地多表写使用本地事务；
- 实际跨服务一致性使用 Seata；
- 文档事务参与者必须与代码调用一致；
- MQ 最终一致性流程不强行改成全局事务。

---

## 7. 配置修改规则

- YAML 注释使用 `#`；
- 禁止使用 `--`；
- 不提交真实密钥或 Token；
- 不提交个人绝对路径；
- Auth 与 Gateway 的 JWT 密钥必须一致；
- 修改 Nacos 配置后必须验证加载和热更新；
- 不假定根目录 `.env` 会自动注入 Spring Boot。

---

## 8. 测试规则

### 8.1 验证优先级

1. 实际启动和复现；
2. 自动化测试；
3. 编译和静态检查；
4. 最小手动请求。

无法执行上一级时才降级，并说明原因。

### 8.2 必须测试的核心场景

- 登录成功和失败；
- Gateway 无 Token、错误 Token、有效 Token；
- 商品查询；
- 创建订单；
- 库存不足；
- 订单和库存分布式回滚；
- 支付结果重复消费；
- 秒杀重复请求；
- 下游服务停止；
- Nacos 热更新。

### 8.3 禁止的测试方式

- 生成大量重复请求充数；
- 使用 `dummy` 请求体；
- 使用固定伪 Token；
- 只断言 HTTP 200 不断言业务码；
- 填写未运行的 P95 或吞吐量；
- 隐藏失败请求。

---

## 9. Git 与提交

Commit 格式：

```text
<type>(<scope>): <subject>
```

示例：

```text
fix(config): correct nacos yaml syntax
fix(order): rollback locked stock on order failure
test(gateway): cover invalid jwt cases
docs(report): record jmeter results
```

每个提交保持单一主题。

---

## 10. 输出要求

完成任务后必须说明：

- 修改了什么；
- 为什么修改；
- 如何验证；
- 实际验证结果；
- 哪些内容未验证；
- 影响了哪些文档。

不得使用“应该可以”作为完成结论。
